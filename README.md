## What Is This?

PharmaChat is a conversational medical assistant that lets you ask natural language questions about medicines вҖ” or simply **photograph a medicine box** and ask about it. The system identifies the medicine from the image, retrieves relevant information from a structured knowledge base, and delivers a clear, concise answer.

Built around a **4-agent LangGraph pipeline**, the system routes each request through specialized agents depending on whether the input is text or an image. All conversations are persisted in **PostgreSQL** and served through a **FastAPI** backend to a clean **React** chat interface.

---

## How It Works

```
Text Input  вҶ’ Rewrite Agent  в”Җв”җ
                               в”ңвҶ’ Retriever Agent вҶ’ Response Agent вҶ’ Answer
Image Input вҶ’ Vision Agent  в”Җв”Җв”ҳ
```

1. **Vision Agent** *(image path only)* вҖ” uses Gemini to identify the medicine name from a photo of its packaging
2. **Rewrite Agent** *(text path only)* вҖ” rewrites the user's query to be more specific and retrieval-friendly, using conversation history to resolve references like "it" or "that medicine"
3. **Retriever Agent** вҖ” searches ChromaDB for the most relevant chunks from the medicine knowledge base
4. **Response Agent** вҖ” reads the retrieved content and generates a concise, conversational answer

## Features

### рҹӨ– Multimodal Multi-Agent Pipeline
- **LangGraph workflow** вҖ” 4 specialized agents connected in a conditional stateful graph
- **Vision input** вҖ” photograph a medicine box or strip; Gemini identifies the medicine automatically
- **Query rewriting** вҖ” resolves follow-up references and improves retrieval accuracy before searching
- **Stateful context** вҖ” full conversation history passed to each agent for multi-turn awareness

### рҹ”Қ Retrieval Pipeline
- **ChromaDB vector store** вҖ” medicine knowledge stored as dense vectors for semantic search
- **HuggingFace embeddings** вҖ” `all-MiniLM-L6-v2` converts text to vectors locally, no API cost
- **Structured JSON knowledge base** вҖ” medicines stored with name, uses, dosage, side effects, and interactions
- **Configurable top-k** вҖ” retrieves top 3 chunks per query for focused, concise answers

### рҹ’ҫ Persistent Conversation History
- **PostgreSQL database** вҖ” every message (text and image path) saved automatically
- **Auto-generated titles** вҖ” conversation titles generated by the LLM from the first message
- **Optimistic UI updates** вҖ” title appears instantly from the first few words, replaced by the AI title after response
- **Full history sidebar** вҖ” all past conversations listed, click any to restore it completely
- **Image persistence** вҖ” uploaded images saved to disk and served as static files; displayed inline in restored chats

### вҡЎ FastAPI Backend
- **RESTful API** вҖ” clean endpoints for conversations, messages, and chat
- **SQLAlchemy ORM** вҖ” typed database models with cascade deletes
- **Static file serving** вҖ” uploaded images served directly from the backend
- **CORS configured** вҖ” ready for local development with the React frontend

### рҹҺЁ React Frontend
- **Clean chat UI** вҖ” user bubbles on the right, assistant on the left
- **Image preview before sending** вҖ” attached image shown in the input bar
- **Inline image rendering** вҖ” images displayed inside chat bubbles
- **Right-click to delete** вҖ” context menu on any conversation in the sidebar
- **Typing indicator** вҖ” animated dots while the assistant is generating a response

## Quickstart

### Prerequisites

- Python 3.10+
- Node.js 18+
- PostgreSQL running locally
- A [Groq API key](https://console.groq.com) (free)
- A [Google Gemini API key](https://aistudio.google.com) (free)

### 1. Clone the repository

```bash
git clone https://github.com/your-username/pharmchat.git
cd pharmchat
```

### 2. Install Python dependencies

```bash
pip install -r requirements.txt
```

### 3. Set up environment variables

```bash
cp .env.example .env
```

Edit `.env`:

```env
GROQ_API_KEY="your_groq_key_here"
GOOGLE_API_KEY="your_gemini_key_here"
```

### 4. Set up PostgreSQL

Create a database and update the connection URL in `database.py`:

```python
Database_URL = "postgresql://postgres:yourpassword@localhost:5432/medicine_assistant"
```

Then create the tables by running the backend once вҖ” SQLAlchemy will auto-create them on startup.

### 5. Ingest the knowledge base

Open `notebook.ipynb` and run all cells. This loads `medicines.json`, splits it into chunks, embeds them with HuggingFace, and stores them in ChromaDB.

> **Re-run this whenever you update `medicines.json`.**  
> Delete `chroma_db/` first to avoid duplicate entries.

### 6. Start the backend

```bash
uvicorn app:app --reload
```

Backend runs at [http://localhost:8000](http://localhost:8000).

### 7. Start the frontend

```bash
cd frontend
npm install
npm run dev
```

Frontend runs at [http://localhost:5173](http://localhost:5173).

## Database Schema

```sql
CREATE TABLE conversations (
    id         SERIAL PRIMARY KEY,
    user_id    INTEGER,
    title      VARCHAR(100),
    created_at TIMESTAMP DEFAULT NOW()
);

CREATE TABLE messages (
    id              SERIAL PRIMARY KEY,
    conversation_id INTEGER REFERENCES conversations(id) ON DELETE CASCADE,
    role            VARCHAR(10),       -- 'user' or 'bot'
    text            TEXT,
    image_path      VARCHAR(255),      -- URL to served image file
    created_at      TIMESTAMP DEFAULT NOW()
);
```

## Design Decisions

**Why conditional routing at the graph entry point?**  
Text and image queries need completely different first steps. A conditional edge from `START` lets LangGraph branch cleanly вҖ” image inputs go straight to the vision agent, text inputs go to query rewriting вҖ” without any `if` statements inside the agents themselves.

**Why Gemini for vision instead of the main LLM?**  
Groq's Llama models don't support image input. Gemini Flash Lite handles vision cheaply and accurately, so the vision agent uses Gemini just for identification, then hands off a text query to the rest of the pipeline.

**Why store image paths instead of base64 in the database?**  
Base64-encoded images are 33% larger than the raw file and bloat the database quickly. Saving images to disk and storing only the URL keeps the database lean and makes images re-loadable in restored conversations.

**Why query rewriting before retrieval?**  
User questions are often short and vague вҖ” "what are its side effects?" doesn't retrieve anything useful. The rewrite agent uses conversation history to resolve pronouns and expand the query into a full, specific sentence before it hits ChromaDB.

**Why optimistic title updates in the frontend?**  
LLM title generation adds a noticeable delay before the sidebar updates. Showing the first 5 words of the user's message instantly makes the UI feel responsive, and the real AI-generated title replaces it silently after the response arrives.
